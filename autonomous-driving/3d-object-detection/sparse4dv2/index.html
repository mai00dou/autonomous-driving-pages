一句话定位：把 Sparse4D v1 的「多帧采样融合（O(T)，显存随帧数线性涨）」升级为 instance bank 递归时序融合（O(1)，恒定开销、可建模任意长历史），并配套三项工程改进（EDA 高效可变形聚合算子、相机参数编码、稠密深度监督），在 nuScenes 上达到 SOTA 且推理更快。

1. 研究背景与动机

1.1 为什么坚持稀疏路线

自动驾驶感知模块的核心是高效融合多视图与时序特征。当下主流是 BEV-based 算法（BEVFormer、BEVDet、BEVDepth、SOLOFusion 等），它们把图像特征变换到 BEV 稠密向量空间再做检测。作者认为 BEV 路线并非最优解，原因有三：
稠密变换无额外收益：把图像特征重排到 BEV 空间只是「换了个坐标系」，没有引入新信息，却增加了模型复杂度。
感知范围/精度/算力难平衡：不同场景（高速、城区、乡村）需要不同的 BEV 网格参数，难以统一权衡。
端到端不友好：稀疏算法产出的 instance 特征（object-centric）更容易与 transformer 等图模型衔接，天然适配端到端自动驾驶。
PETR 系列虽抛弃了 BEV、用 query + 3D 位置编码，但它用的是全局注意力，计算量受图像分辨率显著影响，本质上不算「纯稀疏」算法。因此作者选择 Sparse4D v1（DETR3D 谱系，稀疏 query + 稀疏特征采样，head 计算量与图像分辨率无关）作为 baseline 继续改进。

1.2 v1 的痛点：多帧采样融合，O(T) 显存爆炸

Sparse4D v1 的时序模块有一个显著缺陷：它需要先采样多帧历史数据，再做特征融合（见 Figure 1(a)）。每来一帧，都要把当前帧的 anchor 关键点重新投影到每一个历史帧上、逐帧采样、再用全连接网络融合。
这带来三个问题：
计算复杂度随历史帧数 线性增长 O(T)，推理/训练速度都下降；
GPU 显存随帧数线性上涨（见 Table 1，V100/3090 上实测帧数越多显存越大）；
难以有效引入长时序特征（帧数受显存上限约束，实际只能用约 10 帧）。
v2 的解法：把多帧采样换成 recurrent 递归方式，借鉴 query-based tracker 和 StreamPETR——只传播稀疏的 instance 特征。复杂度从 O(T) 降到 O(1)，既提速降显存，又能融合长时信息。下图（Figure 1）直观对比了两种时序融合范式。

2. 核心问题分析

2.1 相机 3D 检测的三条路线

相机-only 3D 检测的核心是从 2D 图像估计目标的深度/3D 坐标，分三条路线：
单目 3D：病态问题，靠神经网络拟合能力从单图回归相机坐标系深度（FCOS3D、SMOKE 等）。
双目 3D：两个有大重叠角的相机，用特征点匹配 + 方程联合优化或构建 cost volume 求深度（MVSNet）。
多视图 3D：重叠比例介于单目与双目之间，是自动驾驶/机器人最关注的方向，靠多视图 + 时序融合提升深度估计精度。Sparse4D 属于这条路线。

2.3 时序融合的演进脉络

两帧融合起步：BEVDepth、BEVDet4D 缓存上一帧 BEV 特征、warp 到当前帧再 concat；PETRv2 把前后帧特征都作为 cross-attention 的 key。
长时序融合：Sparse4D v1 用 4D 关键点跨多帧采样、FC 融合；SOLOFusion 用最长 16 帧的滑动窗口缓存多帧特征，性能大涨。
递归化（关键转折）：VideoBEV 把 SOLOFusion 的并行融合改成 recurrent 形式降复杂度；StreamPETR 用稀疏 query 在时间维递归传播特征，达到 SOTA。
Sparse4D v2 正是把 StreamPETR 式的 object-centric 递归传播思想，引入到 Sparse4D 的稀疏 deformable 采样框架里——这是它和 StreamPETR 的本质区别：两者都做递归传播，但 Sparse4D 用 deformable 多点采样融合特征，而非 PETR 的全局/局部注意力。

3. 技术架构详解

3.1 整体框架（Overall Framework）

Sparse4D v2 是一个 encoder-decoder 结构（见 Figure 2）。输入三部分：多视图图像、相机参数、上一帧传来的 instance 信息；输出是当前帧精修后的 instances（anchor + 对应 feature），其中一部分高分 instance 被选出来作为下一帧的输入。
整体数据流：
Encoder（图像编码器）：多视图图像经 backbone（ResNet50/101 或 VoVNet-99）+ FPN，提取多视图多尺度特征图 ，其中 是视图数（6）， 是尺度数。
Decoder（稀疏解码器）：由 1 个单帧层（single-frame layer）+ 5 个多帧层（multi-frame layer） 组成。
单帧层含三个子模块：deformable aggregation、FFN、refine/分类输出层；
多帧层在此基础上多两个 multi-head attention：temporal cross-attention（时序交叉注意力，融合历史 instance） 和 self-attention（instance 间交互）。
流转逻辑：先初始化一组 instance（anchor box + feature 向量），用单帧层 refine + 打分，选出前景置信度最高的 instance 作为多帧层输入；多帧层的输入 instance 既来自单帧层，也来自历史帧的输出；多帧层输出即当前帧检测结果，再选高分 instance 喂给下一帧。每层 anchor 数量恒定（不随时序增加），所以推理速度与非时序模型相当。

3.2 Instance 时序传播（核心创新，v1→v2 范式转变）

这是 v2 区别于 v1 的根本所在。在 Sparse4D 里，一个 instance 由三部分表示：
Anchor：结构化信息，描述 instance 的物理状态（3D 检测里就是 3D 框 [x,y,z,w,l,h,sin yaw,cos yaw,vx,vy,vz]），有明确物理含义。
Instance feature：从图像编码器抽取的高阶语义特征。
Anchor embedding：anchor 经一个小的 anchor encoder 编码到高维空间的特征。
这个设计把「图像特征」和「结构化状态」完全解耦，所以加先验知识很方便。时序传播时只需投影 anchor、重新编码 anchor embedding，而 instance feature 保持不变：

工作流概述（细节见下图）：第一帧用单帧 Sparse4D 检测，输出 3D 框 + instance 特征；后续帧把上一帧输出变换到当前帧——feature 不变，anchor 借 ego motion 投影成当前帧 anchor，position embedding 经 anchor encoder 重新编码。每个 decoder 层引入 temporal cross-attention 充分利用时序 instance 特征。时序投影得到的 instance 主要负责 tracklet（已被检测跟踪多帧的目标）；新出现的目标则由单帧单层 decoder 初始化，选高分子集传到后续 decoder。 这样时序模型不增加 anchor 总数，推理速度与非时序模型相当。

3.2.1 代码级拆解：InstanceBank 的 get / update / cache

文件：projects/mmdet3d_plugin/models/instance_bank.py，类 InstanceBank。这是 v2 递归时序传播的代码落点。下面贴关键源码 + 逐段注释。
关键超参（来自 config，v2 论文 §4.2 取值）：
num_anchor=900：每帧 anchor 总数（论文为 900）。
num_temp_instances=600：从历史缓存传播过来的时序 instance 数（论文：600 时序 + 300 单帧 = 900）。
confidence_decay=0.6：历史置信度衰减系数，跨帧衰减历史 instance 的可信度。
default_time_interval=0.5、max_time_interval=2：帧间隔默认值/上限（超出则判定时序失效）。
feat_grad=False：缓存的 instance feature 不回传梯度（配合 detach，保证递归稳定）。
(1) get()：取实例 + ego-motion 投影历史 anchor
def get(self, batch_size, metas=None, dn_metas=None):
    # 初始化一组可学习的 anchor / instance_feature（当前帧新生目标用）
    instance_feature = torch.tile(self.instance_feature[None], (batch_size, 1, 1))
    anchor = torch.tile(self.anchor[None], (batch_size, 1, 1))
    if self.cached_anchor is not None and batch_size == self.cached_anchor.shape[0]:
        # 有历史缓存：算帧间隔，超过 max_time_interval 则 mask 掉（时序失效）
        history_time = self.metas["timestamp"]
        time_interval = metas["timestamp"] - history_time
        self.mask = torch.abs(time_interval) <= self.max_time_interval
        if self.anchor_handler is not None:
            # 构造 temp→cur 的变换矩阵：T_global_inv(当前) @ T_global(历史)
            T_temp2cur = self.cached_anchor.new_tensor(np.stack([
                x["T_global_inv"] @ self.metas["img_metas"][i]["T_global"]
                for i, x in enumerate(metas["img_metas"])
            ]))
            # 把缓存 anchor 用 ego-motion 投影到当前帧（Equation 2 的代码实现）
            self.cached_anchor = self.anchor_handler.anchor_projection(
                self.cached_anchor, [T_temp2cur], time_intervals=[-time_interval],
            )[0]
        ...
    else:
        self.reset()  # 首帧或 batch 变化：清空缓存
        time_interval = instance_feature.new_tensor([self.default_time_interval]*batch_size)
    # 返回：当前帧新 instance + 历史缓存(temp_feature/temp_anchor) + 帧间隔
    return (instance_feature, anchor, self.cached_feature, self.cached_anchor, time_interval)
要点：anchor_projection 就是论文 Equation 2 的实现（在 detection3d_blocks.py 的 SparseBox3DKeyPointsGenerator.anchor_projection，对 center 做 vel × Δt 平移补偿 + R/T 刚体变换，对 yaw/vel 做旋转）。历史 anchor 投影到当前帧后，作为 temp_anchor 喂给 decoder 的 temporal cross-attention。
(2) update()：单帧层后融合历史 instance 与当前新生 instance
def update(self, instance_feature, anchor, confidence):
    if self.cached_feature is None:   # 首帧无历史，直接返回
        return instance_feature, anchor
    ...
    # N = 900 - 600 = 300，从当前帧 instance 里 Top-K 选 300 个新生高分目标
    N = self.num_anchor - self.num_temp_instances
    confidence = confidence.max(dim=-1).values
    _, (selected_feature, selected_anchor) = topk(confidence, N, instance_feature, anchor)
    # 拼接：[600 个历史缓存 instance] + [300 个当前新生 instance] = 900
    selected_feature = torch.cat([self.cached_feature, selected_feature], dim=1)
    selected_anchor  = torch.cat([self.cached_anchor,  selected_anchor],  dim=1)
    # mask 控制：时序有效就用「历史+新生」，否则保留纯当前帧 instance
    instance_feature = torch.where(self.mask[:, None, None], selected_feature, instance_feature)
    anchor = torch.where(self.mask[:, None, None], selected_anchor, anchor)
    return instance_feature, anchor
要点：这是「单帧层负责新生目标、历史缓存负责 tracklet」设计的代码落点——单帧层先打分，update 时只保留 300 个高分新生目标，前面拼上 600 个历史 instance，进入 5 个多帧层。这保证了 anchor 总数恒定为 900，不随时序帧数膨胀。
(3) cache()：detach 截断梯度 + confidence_decay 衰减 + Top-K 写回
def cache(self, instance_feature, anchor, confidence, metas=None, feature_maps=None):
    if self.num_temp_instances <= 0:
        return
    # 关键：detach 截断梯度，递归传播不反传到历史帧（恒定显存、训练稳定）
    instance_feature = instance_feature.detach()
    anchor = anchor.detach()
    confidence = confidence.detach()
    self.metas = metas   # 存当前帧 T_global，供下一帧 get() 投影用
    confidence = confidence.max(dim=-1).values.sigmoid()
    if self.confidence is not None:
        # confidence_decay：历史置信度 × 0.6 与当前取 max，让长期稳定目标维持高分
        confidence[:, :self.num_temp_instances] = torch.maximum(
            self.confidence * self.confidence_decay,
            confidence[:, :self.num_temp_instances],
        )
    self.temp_confidence = confidence
    # Top-K 选 num_temp_instances(600) 个 instance 缓存到 bank，等下一帧
    (self.confidence, (self.cached_feature, self.cached_anchor)) = topk(
        confidence, self.num_temp_instances, instance_feature, anchor)
要点（v2 递归传播三大代码细节）：
detach()：缓存前对 feature/anchor/confidence 全部 detach，梯度不跨帧回传——这是 recurrent 时序「恒定开销 O(1)」的关键，否则会像 v1 一样累积计算图。
confidence_decay=0.6：历史置信度乘以衰减系数再与当前帧 max，既让长期稳定的 tracklet 保持高分被持续传播，又让消失的目标置信度逐帧衰减、最终被 Top-K 淘汰。
Top-K 缓存 600 个：只缓存高分 instance（object-centric memory），恒定开销，与 StreamPETR 的 memory queue 思想一致。

3.2.2 decoder forward 如何调用 bank（recurrent vs v1 多帧采样的代码差异）

文件：sparse4d_head.py，Sparse4DHead.forward。这里把 bank 的 get/update/cache 串进 decoder 的逐层循环里。
decoder 的层序由 operation_order 决定。v2 的配置是：第 1 层（单帧层）= gnn→norm→deformable→ffn→norm→refine（无 temp_gnn）；后 5 层（多帧层）= temp_gnn→gnn→norm→deformable→ffn→norm→refine。关键代码：
# forward 开头：从 bank 取当前帧新 instance + 历史缓存 instance + 帧间隔
(instance_feature, anchor, temp_instance_feature, temp_anchor, time_interval
) = self.instance_bank.get(batch_size, metas, dn_metas=self.sampler.dn_metas)
anchor_embed = self.anchor_encoder(anchor)                  # 当前 anchor → embedding
temp_anchor_embed = self.anchor_encoder(temp_anchor) if temp_anchor is not None else None
for i, op in enumerate(self.operation_order):
    if op == "temp_gnn":           # 时序交叉注意力：当前 instance(query) ← 历史 instance(key/value)
        instance_feature = self.graph_model(
            i, instance_feature, temp_instance_feature, temp_instance_feature,
            query_pos=anchor_embed, key_pos=temp_anchor_embed, ...)
    elif op == "gnn":              # 自注意力：instance 间交互
        instance_feature = self.graph_model(i, instance_feature,
            value=instance_feature, query_pos=anchor_embed, ...)
    elif op == "deformable":       # EDA：4D 关键点稀疏采样多视图多尺度特征
        instance_feature = self.layers[i](instance_feature, anchor, anchor_embed, feature_maps, metas)
    elif op == "refine":           # 输出层：回归 box + 分类
        anchor, cls, qt = self.layers[i](instance_feature, anchor, anchor_embed,
                                         time_interval=time_interval, ...)
        prediction.append(anchor); classification.append(cls)
        # 单帧层结束后立刻 update：把历史 600 + 当前新生 300 融合
        if len(prediction) == self.num_single_frame_decoder:   # =1
            instance_feature, anchor = self.instance_bank.update(instance_feature, anchor, cls)
        ...
# 全部层跑完：把当前帧结果 detach 缓存进 bank，供下一帧递归
self.instance_bank.cache(instance_feature, anchor, cls, metas, feature_maps)
v1（多帧采样）vs v2（递归传播）的代码差异本质：

维度

v1 多帧采样融合

v2 递归传播

历史来源

当前帧 anchor 关键点重新投影到每个历史帧，逐帧 grid_sample

bank 缓存上一帧 Top-K instance（feature+anchor），detach 后传入

融合方式

多帧采样特征用 FC 网络融合

temp_gnn 时序交叉注意力（当前 query ← 历史 key/value）

复杂度

O(T)，存所有历史帧特征图，显存随 T 线性涨

O(1)，只存 600 个 instance，恒定开销

3.3 高效可变形聚合 EDA（Efficient Deformable Aggregation）

Deformable Aggregation（可变形聚合）负责跨多尺度、多视图的特征融合：从多个特征图采样 + 加权求和。论文 Algorithm 1 给出了最基础实现（Basic DA）：
输入：特征图 I={I_s}, 投影 2D 点 P∈R^{K×N×2}, 权重 W∈R^{K×N×S×G}

1. 所有输入存于 HBM（高带宽显存）

2. for 每个尺度 s: 把 Bilinear(I_s, P)∈R^{N×C×K} 写回 HBM

3. stack/reshape 后写 HBM

4. f = f × W ∈ R^{K×N×S×C}，写 HBM

5. 沿 view/scale/point 维 sum → 输出 F

Basic DA 的问题：多次 HBM 读写，且训练时要存大量中间变量做反向传播（ 很大时尤甚），既吃显存又因频繁访存拖慢推理。
v2 的解法：把「特征采样 + scale/view 维加权」封装成 单个 CUDA 算子，一步直接输出多点特征（见 Figure 3）。EDA 的优势：
在 K（关键点）和 C（通道）维完全并行；
单线程计算复杂度仅 ；多视图场景下一个点最多投影到 2 个视图，所以实际单线程复杂度仅 2S；
充分利用 GPU/AI 芯片的并行能力，大幅降显存、提速度。
EDA 是个通用算子，适用于各种多图多尺度融合场景。下图 Figure 3 是论文示意，随后是自绘的 Basic DA vs EDA 对比图。
paper-Figure3-1.png

Figure 3：Efficient Deformable Aggregation。把特征采样 + scale/view 加权封装为单一 CUDA 算子，一步输出多点特征。

自绘对比图：Basic DA（v1，多次 HBM 读写，显存爆炸）vs EDA（v2，单 CUDA 算子，K/C 维并行，单线程复杂度 2S）。EDA 训练显存 -51%，推理 FPS +42%。

3.3.1 EDA 代码落点

文件：blocks.py 的 DeformableFeatureAggregation + ops/deformable_aggregation.py 的 CUDA autograd Function（DeformableAggregationFunction）。
DeformableFeatureAggregation.forward 里有两条路径，由 use_deformable_func 切换：
if self.use_deformable_func:          # v2 EDA 路径：调用 CUDA 算子 DAF
    points_2d = self.project_points(key_points, metas["projection_mat"], metas.get("image_wh"))...
    weights = weights.permute(...).reshape(bs, num_anchor, num_pts, num_cams, num_levels, num_groups)
    features = DAF(*feature_maps, points_2d, weights).reshape(bs, num_anchor, self.embed_dims)
else:                                  # Basic DA 路径（纯 PyTorch，多次访存）
    features = self.feature_sampling(feature_maps, key_points, ...)   # grid_sample
    features = self.multi_view_level_fusion(features, weights)        # 加权
    features = features.sum(dim=2)                                    # 多点求和
DAF（deformable_aggregation_function）是自定义 CUDA autograd Function，forward 调 deformable_aggregation_ext.deformable_aggregation_forward，backward 调对应的 _backward，把采样 + 加权 + 求和融成一个 kernel，避免了 Basic DA 把每步中间张量反复写回 HBM。这就是 Table 2 里训练显存 6328MB→3100MB、推理 FPS 13.7→20.3 的来源。

3.4 相机参数编码（Camera Parameter Encoding, CPE）

v1 里 Deformable Aggregation 的权重由一个全连接层算出，相机参数信息是在训练中隐式地被吸进 FC 层参数里（相当于隐式神经表示 / 对训练集过拟合）。这导致两个问题：
泛化差：交换两张图像的输入顺序时，权重顺序不变，影响感知性能；
对相机参数做大尺度数据增强时，这种隐式表示收敛慢。
v2 的解法：显式把相机参数输入网络——把「输出空间 → 图像坐标空间」的变换矩阵映射成高维特征向量，加到 instance feature 上，再算对应视图的权重。代码落点（blocks.py 的 _get_weights）：
def _get_weights(self, instance_feature, anchor_embed, metas=None):
    feature = instance_feature + anchor_embed
    if self.camera_encoder is not None:    # use_camera_embed=True 时启用 CPE
        # 取 projection_mat 前 3 行(3×4=12 维) → camera_encoder 编码成高维向量
        camera_embed = self.camera_encoder(
            metas["projection_mat"][:, :, :3].reshape(bs, self.num_cams, -1))
        feature = feature[:, :, None] + camera_embed[:, None]   # 加到 instance feature
    weights = self.weights_fc(feature).reshape(...).softmax(dim=-2)...
    return weights
camera_encoder = Sequential(*linear_relu_ln(embed_dims, 1, 2, 12))，即把 12 维（3×4 变换矩阵）映射到 embed_dims。开启 CPE 后 weights_fc 不再需要为每个视图单独输出权重（num_groups*num_levels*num_pts 而非乘以 num_cams）。消融显示：去掉 CPE 会掉 2.0 mAP、4.8 mAOE（朝向估计依赖相机内外参，显式编码能提精度）。

3.5 稠密深度监督（Dense Depth Supervision, DDS）

作者发现稀疏方法在训练早期收敛能力和速度不足。解法：引入多尺度稠密深度估计，用 LiDAR 点云作监督。推理时这个子网络不激活（纯训练辅助）。
代码落点（blocks.py 的 DenseDepthNet）：
对每个尺度特征图用一个 1×1 卷积输出深度，在预定义的「等效焦距」下预测，再乘以「相机实际焦距 / 等效焦距」的比例做尺度对齐；
loss 用 vanilla L1 loss（前景 mask：gt>0 且 pred 非 nan，pred clip 到 [0, max_depth=60]）；
config 里 num_depth_layers=3（监督前 3 个尺度）、loss_weight=0.2、equal_focal=100。
def forward(self, feature_maps, focal=None, gt_depths=None):
    depths = []
    for i, feat in enumerate(feature_maps[:self.num_depth_layers]):
        depth = self.depth_layers[i](feat.flatten(end_dim=1).float()).exp()   # 1×1 Conv + exp
        depth = depth.transpose(0,-1) * focal / self.equal_focal              # 焦距归一化
        depth = depth.transpose(0,-1)
        depths.append(depth)
    if gt_depths is not None and self.training:
        return self.loss(depths, gt_depths)   # 仅训练算 loss
    return depths
在顶层 Sparse4D.forward_train 里：feature_maps, depths = self.extract_feat(img, True, data)，再 output["loss_dense_depth"] = self.depth_branch.loss(depths, data["gt_depth"])。引入 DDS 后，v2 移除了 v1 的 depth-reweight 模块。消融显示：去掉 DDS 会让 mAP/NDS 暴跌 8.5/10.4（训练过程出现梯度崩溃 gradient collapse）——可见 DDS 对稳定训练至关重要。

5. 详细流程图

5.1 单帧推理流程（首帧 / 无历史）

多视图图像 → backbone + FPN → 多尺度特征图；
初始化 900 个可学习 anchor + instance feature（来自 nuscenes_kmeans900.npy 聚类）；
单帧层：gnn（自注意力）→ deformable（EDA 采样）→ ffn → refine（输出 box+cls）；
update() 首帧无历史，直接返回；
5 个多帧层（此时 temp_instance 为 None，temp_gnn 退化）继续 refine；
cache()：detach + Top-K 选 600 个 instance 存入 bank，记录 T_global。

5.2 时序推理流程（后续帧，递归核心）

get()：取新 instance + 历史缓存；用 T_global_inv(当前) @ T_global(历史) 把历史 anchor 投影到当前帧；算帧间隔 + mask；
anchor_encoder 分别编码当前 anchor_embed 与 temp_anchor_embed；
单帧层 refine + 打分（处理新生目标）；
update()：Top-K 选 300 个当前新生 instance，拼上 600 个历史 instance = 900；
5 个多帧层：每层先 temp_gnn（当前 query ← 历史 key/value 时序交叉注意力），再 gnn/deformable/ffn/refine；
cache()：confidence_decay 衰减 + detach + Top-K 写回，供下一帧。

6. 核心创新点

Sparse4D v2 相对 v1 的改进可归为三类（结构 / 训练优化 / 效率）：
创新 1：递归时序融合（结构，最核心）
把 v1 的多帧采样融合换成 instance bank 递归传播。时序信息以 instance（anchor + feature）形式逐帧传递，detach 截断梯度，复杂度 O(T)→O(1)。两大收益：① 大幅提速降显存（Table 1）；② 能融合长时信息，时序带来的性能提升更显著。这是 v2 区别于 v1 的范式转变，思想源自 StreamPETR 的 object-centric 递归，但落在 Sparse4D 的 deformable 稀疏采样框架里。
创新 2：相机参数编码 CPE（结构）
显式把相机内外参（变换矩阵）编码进网络，替代 v1 的隐式 FC 表示，提升对相机参数的泛化与朝向估计精度（消融：+2.0 mAP、改善 4.8 mAOE）。
创新 3：稠密深度监督 DDS（训练优化）
用 LiDAR 点云作多尺度稠密深度监督，1×1 Conv 预测 + L1 loss，仅训练激活。解决稀疏方法早期收敛难、梯度崩溃问题（消融：去掉 DDS 掉 8.5 mAP / 10.4 NDS）。
创新 4：高效可变形聚合 EDA（效率）
把 deformable aggregation 的采样 + 加权融成单 CUDA 算子，训练显存 -51%、最大 batch 3→8、训练时间 23.5h→14.5h、推理 FPS +42%、推理显存 -53%。
单帧层 + 多帧层的分工设计：单帧层专门处理新生目标（选 300 个高分），多帧层处理 tracklet（600 个历史 instance），保证 anchor 总数恒定、推理速度与非时序模型相当。消融（Table 3 Exp.2 vs Exp.3）证明：去掉单帧层、全用多帧层会掉 3.5 mAP / 2.0 NDS。

8. 技术发展时间线

2020 DETR / DETR3D：set prediction，稀疏 query + 3D-to-2D 投影采样，开启 query-based 3D 检测。

2022.03 PETR / BEVFormer / BEVDet4D：PETR 用 3D 位置编码 + 全局注意力；BEVFormer 用 deformable attention 在 BEV 融合；BEVDet4D 两帧 BEV warp+concat 引入时序。

2022.06 PETRv2：前后帧特征都作为 cross-attention 的 key，时序扩展。

2022.10 SOLOFusion：16 帧滑窗长时序融合，性能大涨，但显存高。

2022.11 Sparse4D v1：4D 关键点多帧采样 + FC 融合，纯稀疏路线，但时序 O(T) 费显存。

2023.03 VideoBEV / StreamPETR：VideoBEV 把并行融合改 recurrent；StreamPETR object-centric 稀疏 query 递归传播，SOTA。

2023.05 Sparse4D v2（本篇）：instance bank 递归时序（O(1)）+ EDA + CPE + DDS，速度精度双 SOTA。

后续 Sparse4D v3：在 v2 基础上加 denoising 训练、decouple attention、quality estimation、目标跟踪输出（仓库 main 分支即 v3 集成版）。

9.4 工程注意点

EDA 需编译 CUDA 扩展：use_deformable_func=True 前要先跑 ops/setup.py，否则 DAF 为 None。未编译时会退回 Basic DA（纯 PyTorch，慢且费显存）。
sequential iteration 训练：时序模型按序列顺序迭代（group_in_batch_sampler + with_seq_flag + sequences_split_num=2），保证 bank 缓存的连续性。
detach 是递归稳定的关键：cache 时对 feature/anchor/confidence 全 detach，配合 feat_grad=False，避免计算图跨帧累积。
anchor 初始化：用 nuscenes_kmeans900.npy（对训练集 GT 框 kmeans 聚类得到 900 个先验 anchor）。

10. 局限性与未来方向

局限性：
泛化/鲁棒性待验证：论文坦承 Sparse4Dv2 的泛化能力、场景鲁棒性、长距离检测性能仍需进一步验证。
低分辨率下速度劣势：256×704 输入时 FPS（20.3）不如 StreamPETR（26.7），稀疏采样的优势只在高分辨率才显现。
依赖 LiDAR 深度监督训练：DDS 需要 LiDAR 点云做监督（仅训练阶段），无 LiDAR 数据的场景训练会更难（去掉 DDS 会梯度崩溃）。
依赖准确 ego-motion：anchor 时序投影依赖自车位姿（T_global），位姿噪声会直接影响时序对齐质量。
未来方向：作者希望 Sparse4Dv2 成为新 baseline，并把稀疏路线推广到 HD map 构建、拓扑、轨迹预测、端到端规划等任务（这些方向后续在 v3 及衍生工作中得到延展）。

11. 总结

Sparse4D v2 围绕「提升稀疏多视图时序感知性能」做了三类改进：
结构：时序模块从多帧采样融合 → recurrent 递归传播（instance bank，O(T)→O(1)）；引入相机参数编码。
训练优化：稠密深度监督作为辅助信号，解决稀疏方法早期训练收敛难/梯度崩溃。
效率：Efficient Deformable Aggregation 单 CUDA 算子，训练显存 -51%、推理 FPS +42%。
在 nuScenes 上，Sparse4Dv2 不仅推理速度大幅提升，检测性能也达到 SOTA（test 0.638 NDS，相比 v1 +4.3 NDS）。它是 PETR/DETR3D 稀疏谱系里「递归时序 + 高效采样」的集大成者，与 StreamPETR 同属 object-centric 递归传播范式，但落在 deformable 稀疏采样框架上，为后续 Sparse4D v3（检测+跟踪一体）和端到端稀疏感知奠定了基础。
对马爷 E2E Occupancy 项目的工程启发：
instance bank 递归传播 = 恒定开销的长时序记忆：占据栅格/E2E 任务里，若用 object-centric 或 voxel-centric 的稀疏 query 表示，可借鉴「Top-K 缓存 + detach + ego-motion 投影 + confidence_decay」这套机制，把时序信息以恒定显存累积，避免多帧特征图缓存的显存爆炸。
detach + feat_grad=False 是递归训练稳定的关键：跨帧传播一定要截断梯度，否则计算图累积会 OOM / 训练不稳。
稠密深度监督救收敛：稀疏/query-based 方法早期收敛差是通病，用 LiDAR 稠密深度做训练辅助（推理不激活）能显著稳住训练——E2E Occupancy 若遇到收敛慢，可加类似的稠密几何监督分支。
EDA 思路（融算子降访存）：自定义 CUDA 算子把「采样+加权+求和」融合，是部署落地的关键优化，对车端实时 occupancy 推理有直接参考价值。
anchor/feature 解耦：把结构化状态（可投影、有物理含义）与语义特征解耦，时序传播只动状态、复用特征——这个设计对 occupancy flow / 时序占据预测同样适用。
